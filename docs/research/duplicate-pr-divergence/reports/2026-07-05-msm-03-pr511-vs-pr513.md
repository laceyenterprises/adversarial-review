## Decision

#511 is the stronger survivor. It implements the daemon clean-merge path as a
dedicated decision engine with a typed disposition contract, explicit retry
semantics, prior terminal-failure suppression, and watcher integration tests
that prove daemon ownership does not spawn a closer or hammer. It also absorbed
the final review findings around lease release, transient spawn errors, PR-state
normalization, and missing live-head handling, then passed both Node 20 and Node
22 CI.

#513 is a useful duplicate but not the merge base. It is smaller and more direct,
but its watcher integration changes caused both Node jobs to fail the existing
reviewer-timeout exhaustion fallback tests. Its review also found real issues in
its local daemon helper: floating audit writes, transient preflight failures
outside the retry loop, stdout/stderr diagnostic loss, and loose clean-review
count coercion. Those problems make it a research signal and salvage source, not
the survivor.

## Salvage Folded Into #511

#513's strongest transferable idea was the clean-review count-coercion warning:
a strict zero-finding daemon must not treat `null`, `false`, or an empty string
as `0` merely because JavaScript numeric coercion does. #511 now keeps numeric
zero and numeric strings valid, but treats missing, boolean, empty, or malformed
counts as unknown finding state. That preserves the daemon's fail-closed strict
mode.

Other #513 review findings were already covered by #511's remediated shape:

- Transient preflight/live-gate failures are inside the bounded retry loop.
- Merge subprocess diagnostics combine stderr and stdout at the call site passed
  to `classifyDaemonMergeError`.
- The merge lease is released by an outer post-acquire `finally`.
- The watcher fallthrough path is covered directly by
  `test/watcher-daemon-clean-merge.test.mjs`.

## Research Notes

This pair is a clean example of objective-function divergence. #513 optimized
for a compact helper wired straight into the watcher, but that narrower shape
missed existing fallback behavior and left several partial-failure seams to
review. #511 optimized for an explicit state machine: more code, but a much
clearer ownership contract and stronger evidence that daemon clean-merge cannot
eat the hammer/merge-agent fallback path.

The useful pattern for future daemon-style work is the #511 shape: define the
smallest explicit disposition vocabulary first, prove every caller fallthrough
case, then let the inline daemon own only the states it can prove. The losing
branch was still valuable because it surfaced the strict-count coercion hazard
before merge.
