# AGR-01 Duplicate PR Divergence: PR 373 vs PR 374

Date: 2026-06-20
Repository: `laceyenterprises/adversarial-review`
Ticket: `AGR-01`
Spec: `antigravity-gemini-reviewer@3b374aa861e5`

## Decision

PR #373 is the stronger survivor. It keeps the public CLI in `bin/`, uses a
more stable exported bridge API, writes credentials with a private temp file
plus rename, and avoids the inert `project_id` authorization URL parameter that
appeared in PR #374.

PR #374 should close unmerged after the #373 survivor lands. Its useful ideas
were still salvaged into #373: list-all credential status, callback readiness
before browser launch, clearer callback bind errors, strict permission posture,
and the post-merge action marker.

## Review Finding Convergence

The two independent reviews converged on the same risk clusters:

- Callback host and port behavior: fixed callback port, no fallback, bind/open
  race, and host mismatch.
- Status semantics: `status` performed a live refresh and could rotate tokens
  when an operator expected a read-only check.
- Refresh-token rotation race: both implementations could lose rotated refresh
  tokens under concurrent refreshes.
- Credential persistence risk: rotation persistence errors should not discard a
  valid freshly issued access token.
- CLI and environment contract: both PRs introduced a new auth surface without
  an operator-facing runbook.
- Coverage boundary: #373 placed the CLI in `bin/`, but the lint glob did not
  cover `bin/**/*.mjs`.

This convergence is important: the agents made different implementation choices,
but the adversarial reviews identified nearly identical operational hazards.
For duplicate-agent research, that suggests review comments can be a stronger
signal than raw patch similarity when selecting a survivor.

## Salvaged From PR 374

The final #373 survivor intentionally includes these #374 ideas:

- account enumeration for `agr-auth status` with no account id;
- listener readiness awaited before browser launch;
- callback bind failure propagated before an OAuth browser flow begins;
- explicit credential permission checks on reads;
- post-merge official-record marker in the PR body.

The survivor did not copy #374's in-place credential write, CLI path, inert
`project_id` auth URL parameter, or short-local-part email redaction.

## Remediation Applied To Survivor

The remediated #373 branch adds:

- `localhost` callback host alignment with the registered redirect URI;
- `CALLBACK_SERVER_FAILED` errors that name an occupied callback port;
- a `.ready` callback promise awaited before browser launch;
- read-only `agr-auth status` by default, with `--check-token` for live refresh;
- list-all status mode using credential-file discovery;
- `bin/**/*.mjs` lint coverage;
- strict read-time mode checks for `0700` bridge dirs and `0600` credential
  files;
- in-process single-flight refresh and bounded cross-process lock files;
- rotated refresh-token persistence that preserves the fresh access token on
  persistence failure while warning the caller/operator;
- tests for concurrent refresh, rotation persistence, unsafe permissions,
  credential listing, and busy callback-port startup;
- `docs/RUNBOOK-antigravity-gemini-reviewer.md`.

## Divergence Hypothesis

The two agents appear to have optimized for different local priors:

- PR #373 optimized for packaging and direct fit into the repository's CLI bin
  convention.
- PR #374 optimized for a self-contained auth module with status enumeration and
  readiness flow.

Neither agent independently completed the production hardening story. Each
handled the happy-path OAuth bridge, but both missed at least one important
operator invariant: status must be non-mutating by default, token rotations need
serialization, and callback setup must fail before the browser flow starts.

## Research Takeaways

1. Duplicate PRs should not be merged by first-past-green status. Both PRs were
   test-green, yet both needed final operational hardening.
2. Survivor choice should privilege stable repository fit and safer primitives.
   #373 had the safer file-write primitive and public CLI location, making it a
   better base for final remediation.
3. The loser can still contain high-value design fragments. #374's account
   enumeration and readiness pattern materially improved the survivor.
4. Same-ticket divergences are useful research artifacts. The review comments
   exposed shared blind spots that are not visible from diff stats alone.
5. Future build-pack prompts for OAuth bridge work should explicitly require
   read-only status commands, lock or single-flight refresh rotation, callback
   readiness before browser launch, and governing operator docs.
