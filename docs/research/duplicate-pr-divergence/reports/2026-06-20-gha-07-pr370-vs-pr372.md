## Decision

#370 is the stronger survivor. It already split the adapter work across focused commits, re-applied the reviewer anti-spoof allowlist after the first review, fixed adapter diff fallback after the second review, and documented the optional adapter contract in the SPEC. Its final remaining review findings were non-blocking but addressable, and the branch had a clean merge state.

#372 carried a broader one-commit implementation, but the final review found real fallback contract gaps: malformed adapter output could degrade `fetchState`, mergeability, and diff reads instead of falling back. Its remediation run then stalled before code changes because the mandatory fresh fetch timed out. That left #372 as useful comparison material, but not the best integration base.

## Salvage Folded Into #370

- Added the superproject auto-discovery depth that #372's review called out as a live-service gotcha.
- Hardened adapter binary auto-discovery before token handoff: discovered binaries must live under a trusted `modules/github-adapter` root, be regular executable files, be owned by the current uid or root, and not be group/other writable. Explicit operator env overrides remain available.
- Unified adapter and legacy head/state normalization through a shared helper so adapter-present and adapter-absent hosts do not diverge on state casing or optional fields.
- Removed the string-only review-body fast path. Without structured metadata, adapter review bodies now fall back to the legacy reader so local head/state filtering stays authoritative.
- Added rollout-safe empty discovery behavior: `{pullRequests: []}` from the adapter is cross-checked with Octokit when the fallback client is available.
- Narrowed review-context normalization so that path no longer requires or surfaces full-rollup mergeability fields.
- Aligned the watcher title guardrail `routePR` wrapper with the shared `(title, subject, options)` route signature.
- Added tests for the real superproject discovery path, unsafe discovered binaries, and string-body fallback.
- Documented the trust checks and review-body structured-output contract in the SPEC.

## Final Remediation On Survivor

Final strict-mode remediation on #370 addressed every finding from the 2026-06-20 Claude review:

1. Auto-discovery missed the documented superproject layout.
   - Added `<repo-root>/../../modules/github-adapter/bin/github-adapter`.
   - Added focused coverage that builds the real relative depth.
2. Head/state normalizer divergence risk.
   - Added shared `normalizeHeadAndStateShape` and routed both adapter and legacy head/state paths through it.
3. String review-body fast path bypassed local filtering.
   - Removed the direct return for string-only adapter arrays and forced fallback to the legacy reader.
4. Token-bearing binary selected by bare existence.
   - Restricted auto-discovery to trusted roots and checked file type, owner, executability, and group/other write bits before passing tokens.
5. Empty adapter open-PR list could suppress Octokit discovery.
   - Treated empty adapter discovery as inconclusive when an Octokit fallback client is available.
   - Added regression coverage for adapter `{pullRequests: []}` followed by successful Octokit discovery.
6. Adapter-absent subject reads emitted phantom adapter telemetry and throttle waits.
   - Added an adapter-availability precheck in the subject adapter before wrapping reads in adapter telemetry.
   - Added regression coverage that absent adapters emit only real Octokit/`gh` telemetry.
7. Review-context reused full-rollup mergeability validation.
   - Added a review-context-specific adapter normalizer that returns comments plus metadata and forces `mergeable` / `mergeStateStatus` to `null`.
   - Added coverage for adapter review-context payloads that omit mergeability.
8. `routePR` call shapes diverged across routing modules.
   - Initially made the title-guardrail wrapper accept the shared shape; the final hammer pass removed the two-argument overload entirely and standardized on `(title, subject, options)`.
   - Added focused guardrail routing coverage.
9. `github-api.mjs` still paid adapter throttle on adapter-absent hosts.
   - Added an adapter availability precheck to `runAdapterReadWithTelemetry` so absent adapter reads return before telemetry, action invocation, or throttle wait.
   - Added helper-level coverage for the adapter-unavailable path.
10. Empty adapter review-body arrays were vacuously treated as authoritative.
   - Treated empty structured adapter review-body payloads as inconclusive and fell back to the legacy reader.
   - Added coverage for empty adapter bodies followed by legacy current-head filtering.
11. Guardrail routing still overloaded the second positional argument.
   - Removed the overload and made `watcher-title-guardrails.routePR` use the shared `(title, subject, options)` signature while passing subject through.
   - Updated guardrail tests to use the explicit shared shape.
12. Empty-string adapter diffs could become the reviewed representation.
   - Treated empty adapter diff strings as unavailable so `fetchContent` falls back to `gh pr diff`.
   - Added subject-adapter coverage for empty adapter diff fallback.

## Research Notes

This pair is useful because both branches chose the same high-level adapter-first/fallback-later design, but diverged on how much trust to place in the adapter process. #372 shows the failure mode of broad adapter adoption: catch blocks are easy to add, but malformed-success payloads can still silently degrade state. #370's review cycle shows the stronger pattern for this class of integration: local code must preserve authoritative normalization, head filtering, reviewer identity filtering, credential handoff constraints, adapter-absent telemetry fidelity, and fallback discovery authority even when the external adapter is preferred.
